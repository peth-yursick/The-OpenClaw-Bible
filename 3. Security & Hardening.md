# 3. Security & Hardening

Your OpenClaw has admin access to everything on your machine. Take security seriously. Read this before you start giving your agent tasks.

---

## The Three Rules

### 1. Your Agent Has Access to Everything on Your Computer

Anything you're logged into, any file on disk, any API key in a config file — your agent can access it.

**What this means:**
- Logged into Gmail? It can read/send emails.
- Logged into Twitter? It can tweet from your account.
- Passwords in a text file? It can read them.
- API keys in `.env` files? Full access.

**Mitigation:** Don't be logged into anything on the agent's machine that you don't want it touching.

### 2. Don't Expose Your Agent to the World

If your agent can be talked to by other people, it can be prompt-injected. A prompt injection could make it:
- Leak your API keys
- Send messages on your behalf
- Delete files
- Access your accounts

**What NOT to do:**
- Don't put it in group chats
- Don't let it read tweet replies from strangers
- Don't let it read incoming emails (yet)
- Don't expose the gateway dashboard to the internet

**Keep it private.** Only you should talk to your agent.

### 3. Think Before Every Prompt

Before you give a command, ask: "Will this expose my agent to untrusted input?"

For extra safety, tell your agent:
> "Before executing anything risky, give me a step-by-step plan of what you're about to do."

## Isolation Options

### User Isolation (Minimum)

Create a dedicated OS user for the agent process:
```bash
# macOS
sudo dscl . -create /Users/openclaw-agent
sudo dscl . -create /Users/openclaw-agent UserShell /bin/zsh
sudo dscl . -create /Users/openclaw-agent NFSHomeDirectory /Users/openclaw-agent
sudo mkdir /Users/openclaw-agent
sudo chown openclaw-agent /Users/openclaw-agent
# Don't give it sudo access

# Linux
sudo adduser openclaw-agent
# Don't give it sudo access
```

### Virtual Machines (Recommended)

Run each OpenClaw instance inside a VM for strong isolation.

**Why VMs:**
- Agent is fully sandboxed — can't touch your host OS, files, or accounts
- If it gets compromised, nuke the VM and start fresh
- Each project/agent can have its own VM with only what it needs
- Snapshots let you roll back if something goes wrong

**Options:**
- **UTM** (free, macOS) — Lightweight, Apple Silicon native
- **Parallels** (paid, macOS) — Best performance, seamless integration
- **VMware Fusion** (free personal, macOS) — Solid middle ground
- **VirtualBox** (free, cross-platform) — Works everywhere

**Basic setup:**
1. Create a VM with a minimal OS (Ubuntu Server or macOS)
2. Install only what OpenClaw needs inside the VM
3. No shared folders to host unless needed
4. Use Tailscale inside the VM for secure networking
5. SSH into the VM from your host for management

> For most people starting out: one VM for all OpenClaw work is fine. Split into per-project VMs only when you have sensitive credentials that must stay isolated.

### Docker Sandboxing

Lighter alternative to VMs:

```bash
# Enable Docker sandboxing for all agents
openclaw config set agents.defaults.sandbox.mode "all"
openclaw config set agents.defaults.sandbox.scope "session"
openclaw config set agents.defaults.sandbox.workspaceAccess "ro"
openclaw config set agents.defaults.sandbox.docker.network "none"
openclaw config set agents.defaults.sandbox.docker.memory "512m"
openclaw config set agents.defaults.sandbox.docker.cpus 1
openclaw config set agents.defaults.sandbox.docker.pidsLimit 100

# Verify sandbox configuration
openclaw sandbox explain
```

Docker is less isolated than a full VM (shares the host kernel) but much lighter. Good for when you trust the agent but want guardrails.

## DM Pairing

Only accept messages from verified sources.

**Telegram:** Lock down to your user ID only:
```
# In agent config
TELEGRAM_ALLOWED_USERS=your-user-id-here
```

**Discord:** Use a private server with only you as a member. Don't add anyone else.

## Tool Policy Lockdown

Restrict what tools the agent can use:

### Via OPERATING_CONTRACT.md

```markdown
## Auto-approved actions
- Read files in workspace
- Write files in workspace
- Web search (read-only)
- Summarize content

## Requires approval
- Send any message (Telegram, Discord, email)
- Post to social media
- Make API calls that modify external state
- Install or update packages
- Modify system files

## Forbidden
- Delete files outside workspace
- Access credentials not in the approved list
- Make purchases
- Contact anyone other than the user
```

### Via CLI

```bash
# Deny dangerous tools
openclaw config set tools.deny '["browser", "exec", "process", "apply_patch", "write", "edit"]'

# Allow only safe tools
openclaw config set tools.allow '["read", "web_search", "web_fetch", "sessions_list", "sessions_history"]'

# Disable elevated tool access
openclaw config set tools.elevated.enabled false
```

## File Permissions

```bash
chmod 700 ~/.openclaw
chmod 600 ~/.openclaw/openclaw.json
chmod -R 700 ~/.openclaw/credentials/ 2>/dev/null
chmod -R 700 ~/.openclaw/agents/ 2>/dev/null
```

---

## API Key Management

- Store all API keys in a single `.env` file with `chmod 600`
- Never put keys in command-line arguments (visible in process lists)
- Never put keys in crontab entries
- Rotate keys periodically (see [[Appendix A - CLI Command Reference#Credential Rotation]])
- Use scoped/limited keys where possible (read-only API access when write isn't needed)

## Network Security

### Gateway Binding

```bash
# Bind gateway to localhost only
openclaw config set gateway.bind "127.0.0.1"

# Access remotely via Tailscale: http://100.x.x.x:18789/
# NEVER bind to 0.0.0.0 (exposes to entire network)
```

### Tailscale

Use [Tailscale](https://tailscale.com/) for secure remote access:
1. Install on the agent machine and your personal device
2. Access the agent only through Tailscale's encrypted network
3. No ports exposed to the public internet

## Model Security Note

If using **Kimi K2.5 (Moonshot)** as your brain model: Anthropic models are specifically trained to resist prompt injection attacks. Kimi K2.5's adversarial robustness is less tested.

**Mitigation:** Use tool policy lockdown + Docker sandbox as defense-in-depth. If the agent starts behaving unexpectedly, send `/new` to reset the session.

## Threat Model

| Threat | Likelihood | Impact | Mitigation |
|--------|-----------|--------|------------|
| Prompt injection via group chat | High if exposed | High (full access) | Don't expose to group chats |
| Rogue command execution | Medium | High | Approval queue for risky actions |
| API key leakage | Low if isolated | High | Docker, file permissions, scoped keys |
| Accidental data deletion | Medium | Medium | Backups, approval for deletes |
| Runaway token costs | Medium | Medium | Budget limits, cost tracking |
| Agent posts something stupid | Medium | Medium | Approval queue for public posts |

## Emergency Procedures

### If You Suspect Compromise

```bash
# 1. Kill the agent immediately
openclaw gateway stop

# 2. Check for suspicious processes
ps aux | grep -i "openclaw\|node\|curl\|wget"

# 3. Check recent session activity
ls -lt ~/.openclaw/agents/*/sessions/*.jsonl | head -20

# 4. Look for unauthorized file changes
find ~ -newer ~/.openclaw/openclaw.json -name "*.sh" -o -name "*.py" 2>/dev/null
```

Then: rotate all API keys, review social media accounts, check email sent folder.

### If You See a Billing Spike

```bash
# 1. Stop the agent
openclaw gateway stop

# 2. Check usage
openclaw status --usage

# 3. Review both provider dashboards (Anthropic, Moonshot, OpenAI, etc.)
```

### If the Agent Acts Erratically

```bash
# List active sessions
openclaw sessions list

# Force a fresh session (reset context)
openclaw sessions send --target <session_key> --message "/new"
```

## Weekly Security Audit

Set up a weekly check:

```bash
# Run the built-in security audit
openclaw security audit

# Check if gateway is exposed externally (should fail/timeout)
curl -s --connect-timeout 5 http://YOUR_PUBLIC_IP:18789/
```

Or as a prompt:
> "Every Monday, run a security audit: check file permissions on ~/.openclaw, verify gateway is bound to localhost, confirm no unexpected processes are running, verify all API keys are still valid, check for OpenClaw updates. Report findings on Telegram."

---

## Keep OpenClaw Updated

OpenClaw releases frequently include security fixes.

> "Every Monday, check if there's a new version of OpenClaw available. Compare against the currently installed version. If there's an update, tell me on Telegram what's new (especially security fixes) and ask if I should update."

Don't skip updates — agent security bugs can be critical since the agent has broad system access.

---

## Security Checklist

- [ ] Agent runs in a VM, Docker container, or at minimum a dedicated user
- [ ] No unnecessary login sessions on the agent machine
- [ ] API keys stored in `.env` with 600 permissions
- [ ] Gateway bound to localhost (`127.0.0.1`)
- [ ] Tailscale or VPN for remote access (no public ports)
- [ ] DM pairing (only you can message the agent)
- [ ] Tool policy configured (deny dangerous tools by default)
- [ ] Approval queue for public-facing actions
- [ ] OPERATING_CONTRACT.md defines boundaries
- [ ] Regular backup of workspace
- [ ] Cost limits set on API accounts
- [ ] Emergency kill procedure documented and tested
- [ ] Periodic OpenClaw version check scheduled (weekly)
- [ ] Weekly security audit scheduled
