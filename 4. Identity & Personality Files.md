# 4. Identity & Personality Files

Your agent's behavior is shaped by markdown files in its workspace. These are the "DNA" of your OpenClaw.

---

## Core Files

### SOUL.md (Personality & Voice)

The most important file. Defines who your agent *is*.

```markdown
# Soul

## Identity
You are [name], a [role description].

## Personality
- [Trait 1: e.g., direct and concise]
- [Trait 2: e.g., proactive, doesn't wait to be asked]
- [Trait 3: e.g., thinks in systems, connects dots]

## Communication Style
- [How it talks: casual/formal, emoji usage, humor level]
- [Preferred formats: bullets, headers, short paragraphs]

## Values
- [What it prioritizes: efficiency, accuracy, creativity]
- [What it avoids: fluff, unnecessary confirmations, over-explaining]
```

**Tips:**
- Be specific. "Be helpful" is useless. "Never ask clarifying questions when the intent is obvious — just do it" is useful.
- Include anti-patterns. Tell it what NOT to do.
- Update it over time. If it keeps doing something annoying, add a rule against it.

**If your agent feels boring or generic**, rewrite SOUL.md with personality. Give it opinions, preferences, a sense of humor. Treat it like a character sheet.

See [[Appendix B - Security Templates#Security-Hardened SOUL.md]] for a template that includes critical safety boundaries.

### PRINCIPLES.md (Operating Philosophy)

Higher-level than SOUL.md. Governs decision-making.

```markdown
# Principles

## Decision Making
- Bias toward action. Do things rather than asking permission.
- When uncertain between two good options, pick the faster one.
- If a task will take <5 minutes, just do it. Don't report, just do.

## Quality
- Ship > Perfect. Get it working, then iterate.
- Every output should be immediately usable, not a draft that needs editing.

## Communication
- Be concise. No filler words.
- Lead with the answer, then explain if needed.
- Never say "Great question!" or "Absolutely!" - just answer.
```

### AGENTS.md (Multi-Agent Rules)

Defines how agents coordinate. See [[11. Multi-Agent Coordination]] for details.

Key sections:
- Planning guidelines (think before acting)
- Commit/change guidelines
- File edit conflict resolution
- Which agent handles what domain

```markdown
# Agents

## Planning
- Before any multi-step task, write a brief plan
- Break work into atomic commits
- If a file is being edited by another agent, wait or work on something else

## Domains
- [Agent A]: handles content and social
- [Agent B]: handles code and infrastructure
- [Agent C]: handles research and analysis
```

### USER.md (About You)

Everything the agent knows about you. Gets populated from your conversations but you can also write it directly.

```markdown
# User

## Identity
- Name: [your name]
- Location: [city/timezone]
- Role: [what you do]

## Tech Stack
- [Tools you use]
- [Platforms you're on]
- [Preferred languages/frameworks]

## Goals
- Short-term: [this week/month]
- Long-term: [this quarter/year]

## Preferences
- Communication: [Telegram/Discord/etc]
- Work style: [autonomous vs. check-in-frequently]
- Content: [topics you care about]
```

### MEMORY.md (Long-term Memory)

Accumulated knowledge. The agent reads this at the start of every session.

- Keep it under 200 lines (gets truncated otherwise)
- Link to separate topic files for details
- Update regularly — remove outdated info

### Ideas Library

Use MEMORY.md (or a dedicated `workspace/ideas/` folder) as an ideas library. Whenever you or your agent encounter an interesting concept, tool, project idea, or "what if" thought — capture it.

> "Whenever I share an idea, save it to workspace/ideas/ as a markdown file with: the idea, why it's interesting, potential next steps, and any related links. Tag it by domain (content, product, business, personal). Every Sunday, send me the 3 most promising ideas I haven't acted on yet."

This turns your agent into a second brain that never forgets a good idea and keeps surfacing them at the right time.

### IDENTITY.md (Public-facing Identity)

If your agent posts publicly (tweets, articles), this defines its public persona.

```markdown
# Identity

## Public Name: [name]
## Bio: [one-liner]
## Topics: [what it talks about publicly]
## Tone: [how it sounds in public posts]
## Never: [things it should never say publicly]
```

### TOOLS.md (Available Tools)

Registry of MCP servers and tools the agent can use.

```markdown
# Tools

## MCP Servers
- filesystem: read/write local files
- browser: control web browser
- x-research: search Twitter/X
- firecrawl: scrape web pages

## APIs
- Brave Search: web search
- X API: tweet, read timeline
- OpenAI: coding muscle (Codex)
```

### HEARTBEAT.md (Scheduled Tasks)

Defines what happens during each heartbeat cycle.

```markdown
# Heartbeat

## Every 30 minutes
- Check for pending tasks
- Review message queue
- Update state files

## Daily at 6am
- Generate morning brief
- Send to Telegram

## Daily at 10pm
- Compile daily digest
- Archive completed tasks
- Update MEMORY.md
```

### OPERATING_CONTRACT.md (Boundaries)

Hard rules the agent must follow. Safety rails. See [[3. Security & Hardening]] for the full security context.

```markdown
# Operating Contract

## Must Always
- Ask before spending money
- Ask before sending messages to anyone other than the user
- Log all actions to state/

## Must Never
- Delete files without confirmation
- Post publicly without approval
- Share API keys or passwords
- Access accounts not explicitly authorized
```

## Base Initialization Prompts

When setting up a fresh agent, these prompts bootstrap each system:

| System | What It Does |
|--------|-------------|
| **Brain** | Core reasoning, planning, decision-making, task decomposition |
| **Muscles** | Execution layer — coding, research, content creation |
| **Bones** | Structure — file organization, workspace layout, naming conventions |
| **DNA** | Identity inheritance — how personality traits pass to sub-agents |
| **Soul** | Personality, voice, communication style |
| **Eyes** | Perception — what the agent monitors and notices |
| **Heartbeat** | Scheduled rhythms — cron jobs, periodic checks |
| **Nervous System** | Event handling — how the agent reacts to triggers and signals |

You don't need to use all of these. Start with Soul + Brain + Heartbeat and add others as your setup grows.

## Example: A Direct, No-Fluff Voice

If you want an agent that sounds human, not corporate:

```markdown
## Voice
- Direct, no corporate speak
- Opinionated but backs it up
- Casual tone, like talking to a friend
- Uses analogies and real examples
- No fluff, no "great question!", no "absolutely!"
- Contractions always (it's, don't, can't)
```

Add this to your SOUL.md and adjust to taste. The more specific you are about anti-patterns (what NOT to say), the better the output.

---

## Tips

1. **Don't edit config files manually.** Tell your agent: "I want you to be more concise" and it will update its own SOUL.md.
2. **Review files periodically.** Every few weeks, read through your agent's identity files. Prune anything outdated.
3. **When something annoys you, add a rule.** Agent keeps asking for confirmation? Add "Don't ask for confirmation on routine tasks" to PRINCIPLES.md.
4. **Self-improvement loop:** When your agent does something poorly, tell it: "Pause. Build a skill to handle this better." It will create a new skill and update its own config.
