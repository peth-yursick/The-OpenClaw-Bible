# 2. Core Concepts & Architecture

## How OpenClaw Works

OpenClaw runs as a persistent process on your machine. It:

1. **Listens** for messages (Telegram, Discord, gateway dashboard)
2. **Thinks** using your chosen LLM (Opus, GPT, Kimi, etc.)
3. **Acts** by controlling your computer, browser, files, APIs
4. **Remembers** everything in its filesystem-based memory
5. **Schedules** recurring tasks via heartbeats and cron jobs
6. **Self-improves** by building skills and refining its own config

---

## Brains & Muscles

The most important architectural concept. Don't use one model for everything.

**Brain** = The orchestrator. Decides *what* to do. Needs to be smart.
- Opus 4.6, GPT-4o, etc.
- Handles planning, decision-making, conversation

**Muscles** = Workers. Do the *actual work*. Can be cheaper/specialized.
- Coding: Codex, DeepSeek, local models
- Web search: Brave API
- Social intelligence: X/Grok API
- Summarization: Gemini Flash, Haiku
- Image generation: DALL-E, Flux

### Setting Up Muscles

Tell your OpenClaw:
> "For coding, use the Codex CLI" (give it your OpenAI API key)
> "For finding tweets, use the X API" (give it your X API key)
> "For web search, use the Brave API" (give it your Brave API key)

Now your expensive brain model saves tokens by delegating to cheaper specialized models.

> **Adaptive thinking (Claude 4.6):** The default brain model now adjusts its reasoning depth automatically. Simple questions get fast answers; complex planning triggers deeper analysis. You don't need to prompt for this — it's built in. The tradeoff: occasionally higher token usage on complex tasks. Worth it for the quality improvement.

### Local Models (Advanced)

The endgame: run muscles locally for unlimited free tokens.
- Use Ollama or LM Studio to serve local models
- MiniMax 2.5 locally for coding
- Small models for summarization, classification, routing
- Requires decent hardware (Mac Mini minimum, Mac Studio ideal)

## Heartbeats

Heartbeats are periodic check-ins where your agent evaluates what needs doing.

Default: every 30 minutes, the agent:
1. Checks for pending tasks
2. Looks at scheduled items
3. Evaluates if any proactive work should happen
4. Executes or queues work

You can adjust frequency:
> "Change your heartbeat to every 5 minutes" (for intensive workflows)
> "Change your heartbeat to every hour" (to save tokens)

**Don't edit config files manually.** Tell your agent to change it. It knows how to modify its own configuration safely.

### Lifecycle Status Reactions

Heartbeats now emit status reactions — lightweight signals that other agents and dashboards can observe. When a heartbeat fires, it reports: `idle`, `working`, `blocked`, or `error`.

Useful for mission control dashboards, multi-agent handoffs (check if target is idle before assigning work), and health monitoring (alert if an agent stops emitting).

---

## Cron Jobs

Cron jobs are time-scheduled tasks that run at specific times.

Examples:
- Morning brief at 6am daily
- Social media scan at 1pm
- Nightly research digest at 10pm
- Weekly summary every Sunday

Set them up conversationally:
> "Schedule a task every day at 9am to scan Twitter for [topic] and send me a summary on Telegram"

Your agent handles the cron setup internally.

**Mac tip:** On macOS, `launchd` LaunchAgents are more reliable than cron for scheduled tasks. They're built into macOS, survive sleep/wake cycles, and don't have rate limits. When your agent proposes a cron job, you can ask: "Use a launchd LaunchAgent instead of cron." (Credit: @cathrynlavery)

## Memory System

OpenClaw uses a filesystem-based memory system:

### Short-term: Conversation Context
- Current session messages
- Active task state
- Temporary working data

### Long-term: MEMORY.md + Daily Logs
- Facts about you (goals, preferences, history)
- Lessons learned from past tasks
- Experiment tracking (what worked, what didn't)
- Relationship data (if multi-agent)

### Memory Curation

The agent should periodically:
- Consolidate scattered notes into structured memory
- Remove outdated info
- Promote important learnings to MEMORY.md
- Archive old daily logs

> "Every Sunday, review your memory files. Consolidate learnings, remove outdated info, and update MEMORY.md with anything important."

### MMR Memory Re-Ranking

OpenClaw uses Maximal Marginal Relevance (MMR) to select memories that are both relevant AND cover different aspects of the topic — preventing redundant memories from filling context. Matters most in multi-agent setups with large shared memory. On by default in 2026.3.1+. See [[11. Multi-Agent Coordination#MMR Memory Re-Ranking]] to tune it.

### Tiered Memory with Trust Scoring

Not all knowledge decays at the same rate. Use three tiers instead of a flat file:

**Constitutional** — never expires. Security rules, hard constraints, identity. Getting these wrong once is catastrophic.

**Strategic** — refreshed quarterly. Current projects, creative direction. Stable for months.

**Operational** — auto-archives after 30 days unused. Current bugs, temporary workarounds, session-specific context.

Every entry carries metadata:

```text
- [trust:0.9|src:direct|used:2026-02-22|hits:12] User prefers Telegram over email
- [trust:0.8|src:observed|used:2026-02-20|hits:3|supersedes:old-quirk] Stripe key CAN create products
```

Trust scores: direct user statements = 1.0, agent inferences = 0.7, unverified external sources = 0.5. Hit counts track how often a memory is useful — high-hit memories resist decay. Supersede chains handle contradictions: old versions get archived, not deleted, preventing ghost facts.

The meta-learning: memory itself learns what's important. The system develops a sense of which knowledge matters over time. (Credit: @AtlasForgeAI / @jonnym1ller)

### Active Context Holds

Temporary constraints that shape how your agent interprets everything — not memories, but active filters.

```text
### Product Launch
- **What:** All tasks should prioritize launch prep. Don't start new features.
- **Set:** 2026-03-01
- **Expires:** 2026-03-15
- **Release when:** User explicitly says launch is done
```

The expiry date is critical: without it, holds accumulate into stale frames that distort rather than clarify. If nobody renews a hold, it drops. Think of them as temporary lenses on your agent's priorities. (Credit: @AtlasForgeAI / @jonnym1ller)

### Nightly Extraction (Automated Memory Maintenance)

Every night at 11pm, an automated cron job reviews the day: documents decisions and reasoning, bumps hit counts on used memory entries, and runs the "context is cache, not state" test — could a fresh session reconstruct today from files alone? If not, write what's missing.

Manual synthesis stops happening under load. Automate it.

> "Schedule a nightly cron at 11pm. Review today's sessions: ensure all decisions are documented, bump memory hit counts on entries that were referenced, and check — could a fresh agent session reconstruct today's state from files alone? If not, write what's missing."

(Credit: @AtlasForgeAI / @jonnym1ller)

**Advanced: Spark memory layer**
Flat markdown files are the default memory system, but they're a basic implementation. [Spark](https://spark-coded.xyz) is a memory layer that filters, scores, and distills memories rather than just storing them — and it uses Claude hooks to inject relevant memories into context adaptively. Compatible with OpenClaw. (Credit: @meta_alchemist)

## Nested Sub-Agents

Sub-agents can spawn their own sub-agents, creating multi-level hierarchies (e.g., research coordinator → topic specialists → scrapers). Use when a task naturally decomposes at multiple levels; avoid when a flat list of sub-agents would suffice — nesting adds coordination overhead.

See [[11. Multi-Agent Coordination#Nested Sub-Agents]] for patterns, depth config, and AGENTS.md rules.

---

## ACP Thread-Bound Agents

ACP (Agent Communication Protocol) binds an agent to a specific conversation thread so it maintains persistent context across sessions — no cold starts. Best for long-running projects where continuity matters; overkill for one-off queries.

See [[11. Multi-Agent Coordination#ACP Thread-Bound Agents]] for setup and when not to use it.

## Session Hygiene & Token Management

Tokens cost money. Manage them:

1. **Start fresh sessions** for new topics. Don't let one session balloon forever.
2. **Keep context lean.** Don't load unnecessary files into context.
3. **Use muscles for heavy work.** Route to cheap models when brain-level intelligence isn't needed.
4. **MCP server costs.** Each connected MCP server adds tokens to every request. Only connect what you need.
5. **Skill descriptions matter.** Clear, concise skill descriptions help the router pick the right skill without wasting tokens exploring wrong ones.
6. **Clean up stale sessions.** Old sessions accumulate and waste resources. Use the CLI to prune them:
   ```bash
   openclaw sessions cleanup --older-than 7d
   ```

## Mission Control

A custom dashboard your agent builds for you. Not a pre-built template — it should be custom to your goals and workflows.

Tell your agent:
> "Set up a mission control. This is a custom place for us to build out any tools we need to be more productive. Build it using Next.js and host it locally."

Then iteratively add tools:
- Approvals queue (review tweets/content before posting)
- Task tracker
- Sub-agent status
- Content pipeline
- CRM
- Whatever your workflow needs

Each time you hit a friction point, tell your agent to build a tool for it in the mission control.

## OpenClaw vs Automation Tools

People often ask: why not just use n8n / Make / Zapier?

The difference (from @wickedguro, who runs a social media scheduling SaaS):

| Capability | n8n / Zapier | OpenClaw |
|---|---|---|
| Fixed step-by-step processes | Yes | Yes |
| **Human in the middle** | No | Yes — you can interject mid-task |
| **Local software access** | No | Yes — browser, Puppeteer, desktop apps |
| **Dynamic adaptation** | No | Yes — adapts when data is missing, installs skills |

Automation tools are better for rigid, predictable pipelines. OpenClaw is better when the task is ambiguous, requires judgment, or needs access to things that automation platforms can't touch (like your local browser profile, an app on your computer, or a decision that requires context).

Use both. Route deterministic workflows to n8n/Make. Route anything requiring judgment, adaptation, or local access to OpenClaw.
