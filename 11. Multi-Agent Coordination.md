# 11. Multi-Agent Coordination

Running multiple agents that work together. This is where OpenClaw gets powerful.

---

## Why Multiple Agents?

A single agent has limits:
- One context window at a time
- Can't work on two things simultaneously
- Specialized knowledge bloats one agent's context

Multiple agents let you:
- Parallelize work (researcher + writer + coder)
- Specialize (each agent is expert in its domain)
- Create checks and balances (one agent reviews another's work)

---

## Agent Topologies

### 1. Star (Coordinator + Workers)
One brain agent delegates to specialist workers.
- **Best for:** Most setups. Clear hierarchy, easy to manage.
- **Example:** Coordinator assigns tasks to Coder, Writer, Researcher.

### 2. Pipeline (Chain)
Each agent's output feeds into the next.
- **Best for:** Content pipelines, data processing.
- **Example:** Researcher -> Analyzer -> Writer -> Editor

### 3. Roundtable (Peer)
Agents discuss as equals before taking action.
- **Best for:** Complex decisions needing multiple perspectives.
- **Example:** Three agents debate a strategy, vote, then execute.

### 4. Swarm (Dynamic)
Agents self-organize based on task demands.
- **Best for:** High-volume, varied workloads.
- **Example:** Tasks go into a queue; any available agent picks one up.

## Shared Context via Symlinks

The simplest way to share data between agents: symlink a `shared-context/` folder.

```bash
# Create shared context directory
mkdir -p ~/.openclaw/shared-context

# Symlink into each agent's workspace
ln -s ~/.openclaw/shared-context ~/.openclaw/workspace/agent-a/shared-context
ln -s ~/.openclaw/shared-context ~/.openclaw/workspace/agent-b/shared-context
ln -s ~/.openclaw/shared-context ~/.openclaw/workspace/agent-c/shared-context
```

Now all agents can read/write to the same files:

```
shared-context/
  priorities.md         # Current priorities (all agents read)
  agent-outputs/        # Each agent writes its outputs here
    agent-a-latest.md
    agent-b-latest.md
  handoffs/             # Task handoff queue
  decisions/            # Logged decisions
```

## AGENTS.md File

A global file that all agents reference. Defines coordination rules.

```markdown
# Agents

## Planning Guidelines
- Before any multi-step task, write a brief plan
- Break work into atomic, reversible steps
- Log all plans to shared-context/plans/

## Conflict Resolution
- If two agents need the same file: first to claim it wins
- Claims are logged in shared-context/locks/
- Max lock time: 5 minutes, then auto-release

## Communication
- Agents communicate via shared-context/messages/
- Each message is a markdown file: [timestamp]-[from]-[to].md
- Read and delete after processing

## Handoffs
- To hand off work: write to shared-context/handoffs/
- Include: task description, context, expected output, deadline
```

## Roundtable Conversations

Agents hold structured discussions before major decisions.

**Flow:**
1. A trigger fires (e.g., new strategic question)
2. System creates a roundtable entry in the queue
3. Each agent contributes a perspective (based on their domain)
4. Contributions are synthesized into a decision
5. Decision is logged and executed

**Policy settings:**
- `max_daily_conversations`: 5 (start conservative)
- `enabled`: true/false

This creates emergent behavior — agents influence each other's thinking over time.

## Sub-Agents

Your main agent can spawn temporary sub-agents for specific tasks:

> "Research this topic and give me a summary" -> Spawns a research sub-agent that runs, returns results, then terminates.

Sub-agents:
- Inherit the parent's identity (via DNA files)
- Have limited scope and lifespan
- Don't clutter the main agent's context
- Can run in parallel

---

## Nested Sub-Agents

Sub-agents can now spawn their own sub-agents, creating multi-level task hierarchies.

### When to Use

- **Research sprints:** Coordinator delegates to topic specialists, each specialist spawns scrapers
- **Content pipelines:** Editor spawns writers, each writer spawns researchers for their article
- **Complex analysis:** Strategist spawns analysts, each analyst spawns data collectors

### Depth Configuration

```bash
# Set maximum nesting depth (default: 3)
openclaw config set agents.defaults.subagent.maxDepth 3

# Per-agent override
openclaw config set agents.oracle.subagent.maxDepth 5
```

### Pattern: Research Coordinator

```
Coordinator (Opus) — "Research [topic] comprehensively"
├── Sub-agent A (Sonnet) — "Search Twitter/X"
│   ├── Worker 1 (Haiku) — scrape @user1's threads
│   └── Worker 2 (Haiku) — scrape @user2's threads
├── Sub-agent B (Sonnet) — "Search academic sources"
│   └── Worker 3 (Haiku) — download and extract PDFs
└── Sub-agent C (Sonnet) — "Search competitor blogs"
    └── Worker 4 (Haiku) — scrape and summarize articles
```

**Cost tip:** Route deeper levels to cheaper models. Coordinators need intelligence (Opus/Sonnet). Workers need speed (Haiku/local models).

### Guidelines for AGENTS.md

Define nesting rules in your AGENTS.md:

```markdown
## Sub-Agent Nesting
- Max depth: 3 for research, 2 for content, 1 for admin
- Depth 1 inherits: SOUL.md, OPERATING_CONTRACT.md, full MEMORY.md
- Depth 2 inherits: SOUL.md, OPERATING_CONTRACT.md, task-specific context only
- Depth 3 inherits: OPERATING_CONTRACT.md only (minimal context, maximum speed)
```

## Agent Bindings

Declaratively bind agents to specific channels, threads, or DM conversations.

### CLI Commands

```bash
# List current bindings
openclaw agents bindings

# Bind an agent to a Telegram topic
openclaw agents bind oracle --channel telegram:research-topic

# Bind an agent to a Discord channel
openclaw agents bind flash --channel discord:#coding

# Unbind
openclaw agents unbind oracle --channel telegram:research-topic
```

### Why Bindings Matter

Without bindings, any message to any channel goes to the default agent. With bindings:
- Each channel/topic gets a specialized agent
- The right agent with the right skills handles each conversation
- No routing confusion or context pollution between domains

### Pattern: Topic-Specialized Agents

```
Telegram Group
├── #general → Oracle (coordinator, Opus)
├── #research → Alfred (researcher, Gemini 3.1)
├── #coding → Flash (coder, Sonnet + Codex)
├── #content → Muse (writer, Sonnet)
└── #admin → Butler (ops, Haiku)
```

## ACP Thread-Bound Agents

ACP (Agent Communication Protocol) lets agents maintain persistent context within specific conversation threads.

### How It Works

When bound to a thread via ACP, the agent loads the full thread history on each activation and retains all decisions and context across sessions — no cold starts. Multiple agents can be bound to different threads simultaneously.

### When to Use

- **Long-running projects** — A coding agent bound to a project thread remembers every architectural decision
- **Client conversations** — Each client gets their own thread with dedicated context
- **Research streams** — Ongoing research topics that accumulate knowledge over weeks

### When Not to Use

- One-off tasks (overhead of binding isn't worth it)
- Simple queries that don't need history
- Tasks where fresh context is actually better (avoids anchoring on stale information)

## Lifecycle Status Reactions

Heartbeats now emit lightweight status signals that other agents and systems can observe.

### Status Types

| Status | Meaning |
|--------|---------|
| `idle` | Agent is waiting, no active work |
| `working` | Agent is actively processing a task |
| `blocked` | Agent is waiting on external input or another agent |
| `error` | Something went wrong, needs attention |

### Uses

- **Mission control dashboards** — Real-time agent status overview
- **Multi-agent handoffs** — Check if target agent is idle before assigning work
- **Health monitoring** — Alert if an agent stays in `error` or stops emitting reactions entirely
- **Cost tracking** — Correlate `working` time with API costs

Status reactions are on by default in 2026.3.1+.

## MMR Memory Re-Ranking

When agents share memory, the right memories need to be in context. Maximal Marginal Relevance (MMR) balances relevance with diversity.

### The Problem

Without MMR: Agent searches shared memory for "marketing strategy" → gets 10 memories that all say variations of the same thing → wastes context window on redundancy.

With MMR: Same search → gets 10 memories that each cover a different aspect (pricing, positioning, audience, channels, timing, etc.) → much more useful context.

MMR iteratively selects memories that are relevant AND different from what's already selected — early picks are relevance-driven, later picks are diversity-driven.

### Configuration

No tuning needed for most setups. If you're getting redundant memories, lower the lambda:

```bash
# Diversity weight (0 = pure relevance, 1 = pure diversity, default: 0.3)
openclaw config set memory.mmr.lambda 0.3
```

## Sub-Agent Review Pattern

Most people use sub-agents for parallel execution. A more powerful use: have 3-5 sub-agents *review* Claude's work from completely different perspectives before finalizing.

```
Task: Write an important product announcement

Agent 1 (Devil's advocate): What's wrong with this? What could backfire?
Agent 2 (User perspective): Does this land? Is it clear to someone who doesn't know the context?
Agent 3 (Completeness): What's missing? What questions does this leave unanswered?
Agent 4 (Intent check): Does this actually answer the original question?

Synthesize → publish
```

This catches blind spots a single agent misses. (Credit: @EXM7777)

## Main-Agent-as-Planner Pattern

Don't pollute your main agent's context with execution details. Use it purely for planning and decision-making, then hand off to subagents.

**Pattern:**
1. Plan with main agent — full context, strategic thinking
2. Approve the plan
3. Main agent hands off to a dedicated subagent with the approved plan
4. Subagent executes — knows exactly what to expect, context stays clean

Think of it like plan mode in Claude Code. The main agent stays uncluttered, the subagent focuses purely on delivery.

Add to your `SOUL.md`:
```
spawn subagents for all execution. never do inline work.
```

(Credit: @johann_sath)

## Tips

1. **Start with one agent.** Get it working well before adding more.
2. **Minimum viable multi-agent:** 3 agents (coordinator, executor, observer). 4 core tables. One heartbeat. One worker.
3. **Don't over-engineer.** Two agents talking is enough for most use cases. You don't need six.
4. **Monitor costs.** More agents = more tokens = more money. Use cheap models for workers.
5. **Log everything.** When agents coordinate, debugging is hard. Logs are your lifeline.
6. **Use agent bindings for clarity.** Once you have 2+ agents, bind each to specific channels. It eliminates routing confusion and ensures the right agent handles each conversation.
